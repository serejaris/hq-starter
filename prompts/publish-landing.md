# Промпт — опубликовать лендинг через GitHub + Vercel

Цель — взять один готовый лендинг из проекта и опубликовать его
в интернете через GitHub + Vercel.

Я не инженер. Объясняй простым языком, не вываливай длинные
технические простыни.

## Как ты ведёшь себя в этом диалоге

- **Сначала план, потом действие.** Ничего не меняй, не пушь,
  не разворачивай без моего «ок».
- Один шаг за раз. После каждого жди от меня результата.
- Если что-то упало — разбираешься по ошибке, не «попробуй
  другую команду наугад».
- Не проси у меня пароли, токены или секреты. Любая авторизация
  идёт через официальный браузерный сценарий.

## План

### 1. Найди готовые лендинги
Обычно лежат в `product-landings/`. Найди главный `index.html`
и отдельные страницы продуктов. Покажи мне список.

### 2. Предложи один лендинг для публикации
Если выбор очевиден — предложи лучший вариант с обоснованием.
Если нет — задай 1–2 коротких вопроса.

### 3. Проверь, что выбранный лендинг можно публиковать
- нет приватных заметок и HQ-контекста;
- нет токенов, ключей, паролей;
- все ссылки и локальные файлы работают;
- страница открывается как обычный сайт.

### 4. Подготовь отдельную публичную папку
Только файлы выбранного лендинга. Главный файл = `index.html`.

### 5. Проверь Git и GitHub CLI
`git --version`, `gh --version`, `gh auth status`. Если чего-то
нет или не авторизован — объясни, что сделать (без секретов,
только браузерный OAuth).

### 6. GitHub-репозиторий
Создай новый или используй существующий. Имя — понятное
(например `rfs-landing`).

### 7. Отправь публичную папку в GitHub
`git init` → `add` → `commit` → `push`.

### 8. Подключи Vercel
Если Vercel уже подключён к этому репо — проверь, что деплой
прошёл, и пришли мне ссылку.
Если ещё не подключён — дай точные шаги:
- зайти в vercel.com и войти через GitHub;
- импортировать репозиторий;
- подтвердить, что фреймворк = «Other», корень — публичная папка;
- дождаться первого деплоя.

### 9. Зафиксируй правила в проекте
В `CLAUDE.md` или `README.md` запиши:
- какой репозиторий используется для публикации;
- что публикация идёт через Vercel (auto-deploy на push в `main`);
- что **нельзя** публиковать: приватный HQ-контекст, токены,
  ключи, личные заметки;
- как обновлять сайт: правки → коммит → push → Vercel обновит сам.

### 10. Финальный отчёт
Расскажи мне:
- какой лендинг выбрал;
- какие файлы проверил;
- что опубликовано;
- что **специально не** опубликовано (и почему);
- какой GitHub-репо использован;
- ссылка на живой сайт;
- что добавлено в правила проекта;
- как теперь обновлять сайт.

### 11. Что можно улучшить дальше
Не предлагай всё подряд. Выбери 3–5 действий, самых полезных
для **этого конкретного лендинга**, из:

- усилить CTA;
- улучшить SEO (title, description, h1, структура заголовков);
- Open Graph (картинка + текст для шаринга);
- мобильная версия;
- визуальная иерархия;
- FAQ-блок;
- простая аналитика;
- короткий пост для Telegram / LinkedIn;
- 3 гипотезы, которые можно проверить этим лендингом.

Для каждого пункта дай:
- **что** улучшить;
- **зачем** это нужно (что это даёт);
- **короткий промпт**, который я могу дать агенту следующим
  сообщением.

## Стартуй

Первое сообщение — короткая инициализация (что будем делать)
и список того, что нашёл в `product-landings/`. Сразу после —
один вопрос: какой лендинг публикуем. Дальше идёшь по плану.
